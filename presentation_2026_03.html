01 / 研究背景

为什么这个问题值得做？

超长文档理解是 LLM 落地金融领域的关键瓶颈之一

问题：超长上下文的困境

  • 金融年报、招股书、研报通常 8K - 64K tokens
  • 直接输入 LLM：推理延迟高、成本大、KV Cache 爆炸
  • 截断 / 摘要：丢失关键数字和细节（净利润 -3.2% 变成 "有所下降"）
  • "Lost in the middle" 效应：LLM 对长文本中间部分的注意力显著衰减
  • 金融问答需要精确到具体数字、百分比、日期

价值：高效压缩 + 精确问答

  • 压缩比 128:1 ~ 1024:1（64K → 64 向量）
  • 推理时无需重新编码全文，延迟降低数十倍
  • 教师蒸馏保证压缩前缀 ≈ 完整文档的信息量
  • 冻结 LLM，仅训练 45M 参数（占基座 5.6%）
  • 为金融智能分析提供低成本、高精度的文档理解能力

技术定位

不同于 RAG 的检索 + 拼接范式，我们直接将整篇文档压缩成连续向量作为 LLM 的"记忆"

方法对比
RAG
检索依赖 · 可能遗漏
长上下文 LLM
高质量 · 高成本
DeepCompressor
压缩+蒸馏 · 低成本
核心假设：Perceiver 交叉注意力 + 教师蒸馏能保留足够的细节信息，使固定长度前缀支撑精确金融问答

关键设计决策
冻结 vs 可训练
Qwen3 / FinBERT 全部冻结 (requires_grad=False)，仅训练中间压缩模块 44.93M 参数。Checkpoint 只保存可训练权重。
梯度流设计
冻结 Qwen 前向传播在 torch.no_grad() 内执行。可训练模块（DownProj, QueryInit, Perceiver, UpMLP）在 no_grad 外调用，确保梯度完整回传。
FinBERT 可选：默认 OFF。开启后 Stage B 激活锚点精读，anchor_scores 提供实体级注意力引导，辅助锚点重建损失防止事实丢失。关闭时 Stage B 退化为纯自注意力。
03 / 实际模型架构

无 FinBERT 的压缩流水线

FinBERT 默认关闭 — 系统仅依赖 Qwen 编码 + 问题引导 + 蒸馏完成压缩，Stage B 退化为纯自注意力
与完整架构的区别（FinBERT 关闭后移除）
移除：FinBERT 编码器、NER Head、AnchorAlign、FactDecodeHead
移除：anchor_scores 注意力偏置（Stage A/C 的 cross-attn 无额外 bias）
移除：anchor_embs 锚点精读（Stage B 无 cross-attn）
移除：辅助锚点重建损失
保留的核心机制
保留：Qwen 编码 + DownProj 降维 → byte_array
保留：QueryInit 问题引导初始化（base + question bias）
保留：Stage A/C 两次 cross-attn 读取文档（读-想-重读）
保留：KL 散度 + Hidden MSE 双重蒸馏（数字精度的主要保障）
04 / 训练策略

两阶段训练策略与损失函数设计

Stage 1 解决冷启动，Stage 2 通过多目标损失实现压缩-蒸馏联合优化

Stage 1: NTP 预训练 (Next-Token Prediction)

目标：解决冷启动问题 — 让 Perceiver 输出的前缀携带文档语义信息
数据流：Document → Encoder → DownProj → Perceiver → UpMLP → [prefix] → Decoder → 续写
无教师：不启用 Teacher path，仅优化标准交叉熵续写损失
直觉：如果前缀能让解码器准确续写文档后续内容，说明前缀已经"理解"了文档
LStage1 = CE(Decoder([prefix, x<t]), xt)

Stage 2: QA 微调 + 教师蒸馏

目标：从 Stage 1 checkpoint 恢复，切换到 QA 任务 + 教师蒸馏联合优化
教师路径：同一冻结 Qwen3 读完整未压缩文档 → 输出 logits 和 hidden states
学生路径：压缩前缀 + 问题 + 答案 → 冻结 Qwen3 解码
蒸馏目标：让学生"看压缩前缀"的输出分布 ≈ 教师"看完整文档"的输出分布
Stage 2 多目标损失函数
1. QA 交叉熵损失 w = 1.0
Student 在答案 token 位置的标准自回归损失
LCE = -Σ log Pstudent(at | prefix, q, a<t)
2. KL 输出蒸馏 w = 1.0, T = 2.0
温度缩放后的教师-学生 logit 分布对齐
LKL = T2 · KL(softmax(zT/T) ‖ log_softmax(zS/T))
仅在答案 token 位置计算 · T2 缩放保持梯度量级
3. Hidden State MSE w = 1.0, ramp = 2000 steps
选取四分位层 [7, 14, 21, 27] 的 Q+A 共享位置隐状态对齐
LMSE = ramp(t) · Σl∈{7,14,21,27} MSE(hSl, hTl)
线性 ramp: min(1, step/2000) · 避免早期不稳定
4. 锚点重建损失 w = 0.5 · FinBERT ON 时
FactDecodeHead 从 latent_array 重建 anchor_embs，防止实体信息丢失
总损失
Ltotal = LCE + LKL + LMSE [+ 0.5·Lanchor]
优化器: AdamW (weight_decay=0.01) 调度器: Cosine Warmup: 1000 steps Max Grad Norm: 1.0 梯度累积: 4 steps (等效 batch=16) 混合精度: bf16 / fp16 (可选)

10 / 困难与反思

已解决的工程难题

从冷启动到梯度流，开发过程中遇到并解决的核心技术挑战

◆ 冷启动问题

随机 Perceiver 输出对解码器是噪声 → 解码器学会忽略前缀。
解决：两阶段训练。Stage 1 NTP 预训练迫使前缀携带文档信息。

◆ 教师-学生序列长度不一致

Teacher 输入 64K token，Student 前缀仅 64 向量。
解决：仅在 Q+A 共享 token 位置计算 Hidden MSE，选取 [7,14,21,27] 四分位层。

◆ 冻结/可训练模块梯度断裂

torch.no_grad() 上下文可能意外覆盖可训练模块。
解决：严格区分 no_grad 作用域 + Exp 2 验证每层梯度非零。

◆ 64K Teacher 显存开销

Teacher 读取完整文档的显存可能 OOM。
解决：预计算并缓存 Teacher 的 logits 和 hidden states。

11 / 深层反思

模型的潜在结构性问题

超越工程 bug，思考方法层面的根本性局限

⚠ 与基座模型的深度绑定 — 迁移性为零

整个压缩器围绕 Qwen3 的表示空间、词表、层结构设计。DownProj/UpMLP 学习的是 Qwen 特定维度间的映射；蒸馏对齐的是 Qwen 特定层 [7,14,21,27] 的 hidden states。换基座模型 = 从头训练。两个 LLM 的表示空间不对齐，DeepCompressor 本质上是 Qwen3 的专属外挂，不是通用压缩器——限制了方法的学术贡献和部署灵活性。

⚠ 具体数值信息的必然丢失

金融问答的核心是精确数字（"净利润 -3.2%"、"2024Q3 营收 847.6 亿"）。但信息经过连续向量瓶颈后，数字会从离散 token 变为浮点坐标中的隐式编码。关键问题：

(1) 数字 token 在 Qwen 词表中是组合编码的（"847.6" 可能被拆为 "8","47",".","6"），Perceiver 需要在连续空间中保持这种组合关系的精确性——梯度下降没有机制保证这一点；
(2) 一篇 64K 文档中可能包含数百个不同数字（财务表格、时间序列），压缩到 64 个向量后，每个向量需要"记住"约 5-10 个精确数字——远超单个连续向量的有效信息承载量；
(3) 相似数字（"3.2%" vs "3.3%"、"2024" vs "2025"）在连续空间中的距离极小，但语义差异巨大，模型很难学到足够精细的区分。

这可能是该方法最根本的瓶颈——即使整体 F1 不错，Exact Match 在数字密集型问题上可能很低。

⚠ 连续压缩的数学困难

从信息论角度：64 个 1024 维 float32 向量 ≈ 256KB，64K token 原文 ≈ 128KB。表面上容量够，但：

(1) 蒸馏损失只优化分布相似性（KL 散度），不直接优化精确 token 复现；
(2) 压缩比 1024:1 时，存在率失真理论的基本限制——某些细节必然丢失；
(3) 连续空间中精确编码离散符号效率极低，大量维度被浪费在"平滑"上。

⚠ 训练效率瓶颈

单样本需三次 Qwen 前向传播（编码文档 + 解码生成 + Teacher 路径）。64K 输入的显存和计算开销巨大。Teacher 可缓存，但Student 编码无法避免完整前向传播。

⚠ 缺乏理论保证

Perceiver 压缩 + 蒸馏对齐的组合是纯经验性设计——没有理论保证压缩后的前缀能保留多少信息。蒸馏损失收敛不等于信息无损。三阶段 Perceiver 的设计更多是直觉驱动而非数学推导。
